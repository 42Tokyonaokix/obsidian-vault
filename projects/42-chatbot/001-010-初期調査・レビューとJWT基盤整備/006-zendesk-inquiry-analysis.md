## 概要

Zendesk問い合わせデータ339件（45 intent）を分析し、チャットボットによるCS業務軽減のための機能実装優先度とRAG評価テスト戦略を策定した。既存の `inquire_features.md`（A-1〜A-4）のカバー範囲は全体の約28%に留まり、残り72%のintentに対する対応方針を整理した。

### 4. テスト戦略

#### 4.1 RAG評価テストの設計方針

現在の `rag_eval_results_v2.csv` は40件程度だが、Zendesk実データから**実際のユーザー質問をそのまま使う**ことで、現実的な評価が可能になる。

**テスト分類:**

| テストタイプ | 目的 | 評価基準 |
|-------------|------|---------|
| **回答品質テスト** | RAGが正確な情報を返すか | expected_answerとの意味的一致（LLM-as-judge） |
| **エスカレーション判定テスト** | bot完結 vs Zendesk起票の判定が正しいか | 正しいアクション選択率 |
| **メタデータフィルタテスト** | 約款/地域で適切なチャンクが返るか | 返却チャンクのvoltage_type/area一致率 |
| **ハルシネーション検出テスト** | 知識にない質問に対して「分からない」と言えるか | 誤情報生成率 |

#### 4.3 テスト実装手順

1. **Phase A テスト（T-001〜T-015）**: ナレッジチャンク追加前に「現状スコア」を計測 → チャンク追加後に「改善スコア」を計測（before/after比較）
2. **Phase C テスト（T-101〜T-112）**: 機能実装後の受け入れテストとして使用
3. **メタデータフィルタテスト（T-201〜T-205）**: 002-rag-metadata-filtering タスクのST-6で使用
4. **エスカレーション判定テスト（T-301〜T-305）**: エスカレーション精度の定量評価

## 決定事項

- Zendesk 339件のintentを5つのTierに分類し、対応方針を確定
- Phase A（ナレッジ拡充）が最もROIが高い — 開発コストほぼゼロで全体の31%をカバー
- RAG評価テストはZendeskの実質問をそのまま使用する方針
- A-1〜A-4の実装順序は A-2 → A-1 → A-4 → A-3（ROI順）
