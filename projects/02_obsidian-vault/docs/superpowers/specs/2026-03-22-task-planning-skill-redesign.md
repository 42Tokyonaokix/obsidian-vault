# タスクプランニングスキル再設計

## 背景・目的

`01-tasks-add` と `01-tasks-add-auto` が Obsidian に出力するタスクファイルの情報量が少なすぎる。現状は「概要 + チェックボックス」のみで、プランニング過程（ユーザーとの対話内容、エージェント間の議論）が消えてしまう。結果として、タスクファイルを見たエージェントがそのまま実装に着手できない。

両スキルを「タスク列挙」から「実装着手可能なプランニング」に再定義し、Obsidian への出力を大幅に拡充する。

## スコープ

### やること

- `01-tasks-add` のテンプレートを拡張し、ユーザーとの会話内容をすべて構造化して出力する
- `01-tasks-add-auto` のテンプレートを拡張し、エージェント間の議論をすべて出力する
- `01-tasks-add-auto` のフローを直接コミットからPRベースの承認フローに変更する
- 両スキルの description とトリガーワードを更新する
- ルータースキル `01-tasks` の description を更新する

### やらないこと

- スキル名のリネーム（ルーターとの整合を維持）
- 他スキル（list, update, daily, execute）の変更
- frontmatter スキーマの変更
- 3エージェント（PM/Architect/Critic）のプロンプト変更

## 設計

### 01-tasks-add の改修

**位置づけ:** 対話型プランニング + タスク追加

**原則:**
- ユーザーとの会話で出た内容をすべて構造化して出力する
- 会話に出ていない推測は書かない
- 対話中にリアルタイムで承認されているので直接コミット（従来通り）

**対話フローの変更:**
- 従来通り1問ずつ質問して明確化
- 会話中に出た全ての情報（背景、スコープ判断、技術的議論、却下した案など）を追跡し、最終的にテンプレートに流し込む

**新しいタスクファイルテンプレート:**

```markdown
---
title: "..."
date: YYYY-MM-DD
project: project-name
status: todo
progress: 0/N
priority: high
tags: []
---

## 背景・目的

なぜこのタスクが必要になったか。ユーザーが語った動機・課題をそのまま記録する。

## スコープ

### やること
- 会話で合意した作業範囲

### やらないこと
- 会話で明示的に除外したもの

## 設計判断

会話の中で検討・決定した設計上の選択。
- 決定事項とその理由
- 検討したが却下した代替案（会話で出た場合）

## タスク

- [ ] サブタスク1
- [ ] サブタスク2

## 各タスクの詳細

### サブタスク1: {名前}

- **目的**: 何を達成するか
- **やること**: 具体的な作業内容
- **技術的ポイント**: 会話で議論した技術的な注意点
- **完了条件**: 何をもって完了とするか

（会話で議論した粒度に応じて項目は増減する）

### サブタスク2: {名前}
（同上）

## 前提条件・依存関係

会話で確認した前提や依存。

## 補足

会話で出たがセクションに収まらない情報（参考URL、関連する既存実装の話など）
```

各セクションは会話で該当する内容がなければ省略可。

### 01-tasks-add-auto の改修

**位置づけ:** AIエージェントによるプランニング + PRベースの承認

**原則:**
- 3エージェント（PM/Architect/Critic）の議論内容をプランとして Obsidian に出力する
- エージェントの生の思考過程（判断ログ、技術調査メモ）もそのまま記録する
- ユーザーが内容を確認していないため、PRで承認を待つ

**3エージェントのプロンプト:** 変更なし。現状のまま。

**Step 4（ターミナル提示）の変更:**
- 現状の「詳細 → 戦略 → 要点」のターミナル表示を廃止
- 代わりにPRのURLとタイトル・概要のサマリーだけ提示
- 詳細はPR上で確認してもらう

**Step 5（ユーザーレビュー）の変更:**
- ターミナルでの「承認/修正/再ブレスト」を廃止
- PR上でレビュー。修正が必要なら会話で指示 → 3エージェントパイプラインを再実行し、結果を同じブランチに新コミットとして追加
- マージされたら完了。実装着手は別途ユーザーが指示する

**PRフロー:**
1. `git pull` で最新化
2. ブランチ `task/{project}/{slug}` を作成（slug はタスクファイルのファイル名 slug と同一。例: `task/42-chatbot/007-a1-capacity-change-navigator`）
3. タスクファイルを作成してコミット
4. `gh pr create --title "plan({project}): {slug}" --body "{概要セクションの内容}"` でPRを出す
5. ユーザーに PR URL を提示して完了

**新しいタスクファイルテンプレート:**

```markdown
---
title: "..."
date: YYYY-MM-DD
project: project-name
status: todo
progress: 0/N
priority: high
tags: []
---

## 背景・目的

ユーザーのアイデアと確認した目的をそのまま記載。

## 要件定義

PM Agent が整理した要件。
- 目的と定義理由
- スコープ（やること / やらないこと）と境界線の判断理由
- 成功基準

## 技術検討

Architect Agent の技術的分析。
- 技術的実現性と難易度の判断根拠
- 推奨アプローチと比較検討した代替案
- リスクと対策
- 関連する既存コードやファイルパスの引用

## 議論サマリー

Critic Agent による PM・Architect の見解の突き合わせ。
- 合意点
- 意見が分かれた論点と各者の主張
- 裁定と根拠

## 設計判断

議論を経て確定した設計上の選択。
- 採用したアプローチとその理由
- 却下した代替案とその理由

## タスク

- [ ] サブタスク1
- [ ] サブタスク2

## 各タスクの詳細

### サブタスク1: {名前}

- **目的**: 何を達成するか
- **やること**: 具体的な作業内容
- **技術的ポイント**: 難易度、使う技術、注意点
- **対象ファイル**: 変更が必要なファイル（Architect が特定した場合）
- **前提/依存**: 他タスクとの関係
- **完了条件**: 何をもって完了とするか

### サブタスク2: {名前}
（同上）

## 前提条件・依存関係

エージェントが特定した前提や依存。

## 補足

エージェントの議論で出たがセクションに収まらない情報（参考URL、関連する既存実装の話など）

## PM 判断ログ

PM Agent の生の思考過程をそのまま記録。

## アーキテクト技術調査メモ

Architect Agent の生の調査・気づきをそのまま記録。

## Critic 議論ログ

Critic Agent の生の統合・裁定過程をそのまま記録。「議論サマリー」セクションは Critic の出力から構造化した要約であり、本セクションは生の出力の全文保存。
```

### メタデータの変更

**`01-tasks-add` の description（before → after）:**
```
# before
description: Use when the user wants to create or break down a new work item. Triggers on "やること追加", "分解して", "具体化して", "どう進めればいい", "タスクに落として", "break down", "clarify steps".

# after
description: Use when the user wants to plan and create a new work item through conversation. Triggers on "やること追加", "分解して", "具体化して", "どう進めればいい", "タスクに落として", "break down", "clarify steps", "プランニング", "計画して".
```

**`01-tasks-add-auto` の description（before → after）:**
```
# before
description: Use when the user wants to auto-brainstorm and create tasks with AI agents. Triggers on "タスク自動追加", "自動ブレスト", "AI でタスク作って", "auto task", "brainstorm task", "タスクをAIで具体化", "アイデアをタスクにして".

# after
description: Use when the user wants AI agents to plan and create tasks through automated brainstorming. Triggers on "タスク自動追加", "自動ブレスト", "AI でタスク作って", "auto task", "brainstorm task", "タスクをAIで具体化", "アイデアをタスクにして", "AIプランニング".
```

**`01-tasks` ルーティングテーブルの変更:**

| User Intent | Route To | 変更 |
|---|---|---|
| タスク追加・作成・分解・プランニング | 01-tasks-add | 「プランニング」追加 |
| タスク自動追加・AIブレスト・AIプランニング | 01-tasks-add-auto | 「AIプランニング」追加 |

ルータースキルの triggers にも `"プランニング"`, `"計画して"`, `"AIプランニング"` を追加する。

### 他スキルへの影響

- `01-tasks-list`, `01-tasks-update`, `01-tasks-daily`: frontmatter スキーマ変更なし。影響なし
- `01-tasks-execute`: タスクファイルの情報量が増えるため、実装着手しやすくなる。変更不要
