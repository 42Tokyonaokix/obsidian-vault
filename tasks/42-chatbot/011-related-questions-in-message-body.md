## 背景・目的

ユーザーから「返信の選択肢がメッセージに含まれてしまっている」という報告があった。`SingleReactOrchestrator.astream()` が LLM の `agent` ノードからのトークンを無条件に `token` イベントとして送出しており、`[RELATED_QUESTIONS]` デリミタ以降の関連質問テキストもメッセージ本文としてフロントエンドに流れている。ChipRow に表示されるべき選択肢テキストがメッセージ本文に表示されるため、回答の読みやすさが著しく損なわれる。

## 要件定義

- **目的**: `SingleReactOrchestrator` のストリーミング中に関連質問テキストがメッセージ本文に混入する問題を解消する。バックエンドでの根本修正とフロントエンドでの防御層の2段構えで対応する。
- **スコープ**:
  - やること: `SingleReactOrchestrator.astream()` のトークン送出時にデリミタ検知・抑止ロジックを追加
  - やること: フロントエンド `useConversation.ts` に防御的サニタイズを追加
  - やること: バックエンド・フロントエンド双方のテストを追加
  - やらないこと: `relatedQuestions` の表示UI（ChipRow）自体のデザイン変更
  - やらないこと: `MultiAgentOrchestrator` の変更（こちらは `add_related_questions_node` で `ui_resource` に格納する方式のため問題なし）
  - やらないこと: `SingleReactOrchestrator` のプロンプト設計を `MultiAgentOrchestrator` 方式に統一する大規模リファクタ（別タスクとして検討すべき規模）
- **成功基準**:
  1. ボットの回答テキストに `[RELATED_QUESTIONS]` デリミタや関連質問テキストが含まれない
  2. 関連質問テキストは ChipRow にのみ表示される
  3. 関連質問がない回答でも正常にメッセージが表示される
  4. 既存の `test_chat_stream.py`、`useSSE.test.ts`、`conversation.test.ts` が引き続き pass する

## 技術検討

### 原因の特定

2つのオーケストレーターで挙動が異なる:

**SingleReactOrchestrator** (`src/app/services/agent/single_react.py`):
- LLM のシステムプロンプトが `[RELATED_QUESTIONS]` デリミタを使って返信選択肢を本文末尾に埋め込む設計
- `astream()` メソッドは `stream_mode=["messages","custom"]` で LLM のトークンをリアルタイムに流す（L294-311）
- しかし `_extract_related_questions()` の除去処理（L404-430）はストリーミング完了後の `aget_state()` で取得した最終メッセージに対してのみ行われる
- **問題**: トークンストリーミング中は `[RELATED_QUESTIONS]` を含む生テキストがそのまま `token` イベントとして流れ、フロントエンドはこれをメッセージ本文として蓄積する

**MultiAgentOrchestrator** (`src/app/services/agent/multi_agent/orchestrator.py`):
- `add_related_questions_node` が別途 LLM 呼び出しで関連質問を生成し `ui_resource["related_questions"]` に格納（L489）
- `token` イベントには混入しない → こちらでは問題なし

**フロントエンド** (`widget/src/hooks/useConversation.ts`):
- `onToken` コールバック（L157-179）はトークンを純粋に蓄積するだけで、デリミタのフィルタリングを一切しない
- `onRelated` コールバック（L182-195）は呼ばれた時点で `relatedQuestions` を設定するが、既に蓄積された本文テキストからデリミタ以降を除去する処理がない

### 推奨アプローチ

**バックエンド（根本修正）**: `SingleReactOrchestrator.astream()` のトークンストリーミング処理で、`[RELATED_QUESTIONS]` デリミタを検知した時点で以降のトークン送出を停止する。`full_answer` バッファを監視し、デリミタの部分一致もバッファリングで検知する。

**フロントエンド（防御層）**: `useConversation.ts` の `onRelated` コールバック付近で、最後のアシスタントメッセージの `content` から `[RELATED_QUESTIONS]` 以降を除去する防御的サニタイズを追加。LLM がデリミタ形式を微妙に変えた場合のフォールバックとして機能する。

### 却下した代替案

- **フロントエンドのみでの事後フィルタリング**: 根本原因を修正せず表層で隠すだけ。バックエンドの問題を温存する。
- **`SingleReactOrchestrator` のプロンプト設計を `MultiAgentOrchestrator` 方式に統一**: 根本的な解決だが、変更範囲が大きく（プロンプト・`_extract_related_questions` 廃止・`astream` 再設計）、別タスクとして切り出すべき規模。

### リスク

- **トークン分割境界**: LLM がトークン単位でストリームするため、`[RELATED_QUESTIONS]` 文字列が複数の `token` イベントに分割される可能性がある。バッファリングロジックで部分一致を検知する実装が必要。
- **LLM のプロンプト遵守率**: LLM がデリミタ形式を正確に守らないケース（`[RELATED_QUESTION]` typo 等）への対応として、フロントエンドの防御層が有効。

## 議論サマリー

### 論点と裁定

**論点1: 原因は特定済みか**

- **裁定**: アーキテクトの分析を採用。コード上で混入経路が明白であり、再現フェーズは不要。

**論点2: バックエンドのみか、フロントエンドにも防御層を入れるか**

## PM 判断ログ

バグ2（選択肢の本文混入問題）は、「可能性がある」という記述からもわかるように、原因がまだ特定しきれていない。

コードを読んで重要なことがわかった。`useSSE.ts` の SSE イベント処理を見ると、`token` と `related` は完全に別イベントとして独立しており、`token` コールバックのストリームに `related` のテキストが混入する経路がフロントエンドには存在しない。つまりフロントエンドのロジック上、「関連質問テキストがメッセージ本文に追記される」パスはない。問題があるとすれば、バックエンドが `token` イベントとして `related` の内容をストリームしている場合か、あるいは「問題が起きている」と報告されているが再現条件が不明確な可能性がある。バグ2の修正前に、まず「どの層で混入しているか」を特定するフェーズが必要だと判断した。

優先度は medium（再現確認後は high）。コードベースを読んだ限りフロントエンドに明確な混入経路が見当たらず、再現条件が不明確な状態。ただし再現が確認されれば、回答品質に直接影響するため high に引き上げる。

却下した選択肢:
- フロントエンドで `relatedQuestions` のテキストをメッセージ本文から事後的にフィルタリングする方式: 根本原因を修正せずに表層で隠すだけであり、バックエンドの問題を温存する。
- 原因特定なしに両層を一度に修正する方式: 変更範囲が広くなりリグレッションリスクが上がる。

## アーキテクト技術調査メモ

原因の特定: 2つのオーケストレーターで挙動が異なる。

**SingleReactOrchestrator** (`src/app/services/agent/single_react.py`):
- LLM のプロンプト（`system_prompt.md` L40-41）が `[RELATED_QUESTIONS]` デリミタを使って返信選択肢を本文の末尾に埋め込む設計
- `astream` メソッドは `stream_mode=["messages","custom"]` で LLM のトークンをリアルタイムに流している（L294-311）
- しかし `_extract_related_questions()` の除去処理（L404-430）はストリーミング完了後の `aget_state()` で取得した最終メッセージに対して行われる
- **問題**: トークンストリーミング中は `[RELATED_QUESTIONS]` を含む生テキストがそのまま `token` イベントとして流れ、フロントエンドはこれをメッセージ本文として蓄積する。除去処理は行われない

**MultiAgentOrchestrator** (`src/app/services/agent/multi_agent/orchestrator.py`):
- `add_related_questions_node` が別途 LLM 呼び出しで関連質問を生成し `ui_resource["related_questions"]` に格納する（L489）
- `final_response` には `[RELATED_QUESTIONS]` デリミタは含まれない
- `token` イベントとして流れるのは `final_response` のみなので、こちらではバグが再現しない

**フロントエンド側**: `useConversation.ts` の `onToken` コールバック（L157-179）はトークンを純粋に蓄積するだけで、デリミタのフィルタリングを一切しない。`onRelated` コールバック（L182-195）は呼ばれた時点で `relatedQuestions` を設定するが、既に蓄積された本文テキストからデリミタ以降を除去する処理がない。

推奨アプローチ:
- 案A（採用）: バックエンドのストリーミングループ内でデリミタを検知し、以降のトークンを送出しない。`full_answer` をバッファとして使い既存変数を再利用できる。
- 案B（補助として推奨）: フロントエンドの `onRelated` コールバックで `setMessages` 時にメッセージ本文から `[RELATED_QUESTIONS]` 以降を除去する防御的フィルタ。案Aとの2重防衛として有効。
- 案C（却下）: `SingleReactOrchestrator` のプロンプトを書き換えて `[RELATED_QUESTIONS]` デリミタ形式を廃止し、MultiAgentOrchestrator と同様の別ノード方式に統一する。根本的な解決だが、変更範囲が大きく（プロンプト・`_extract_related_questions` 廃止・`astream` 再設計）、別タスクとして切り出すべき規模。

リスク: `[RELATED_QUESTIONS]` デリミタがストリームの途中で分割されて届くエッジケース。LLM がトークン単位でストリームするため、文字列が複数の `token` イベントに分割される可能性がある。バッファリングロジックは部分一致でデリミタを検知する実装が必要。
