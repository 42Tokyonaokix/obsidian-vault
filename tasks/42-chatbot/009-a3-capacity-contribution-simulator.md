## 概要

Zendesk問い合わせ34件（キーワード直接ヒット分）に対応する容量拠出負担金の説明・シミュレーション機能を実装する。

### Zendesk実データ分析結果（2026-03-22調査）

問い合わせの内訳:
- **解約後の請求（20件・59%）**: 「解約したのに請求が来た」「最後の請求はいつか」— 最多パターン
- **値上がり・来年度単価（7件・21%）**: 年度切替時に集中。「2026年4月から値上がりすると聞いた」「来年度の想定単価を教えてほしい」
- **金額・計算（4件・12%）**: 精算額の内訳確認、複数拠点の案内タイミング
- **制度説明・請求タイミング（3件・9%）**: 「継承したが今回の請求にないのはなぜ」等

### 現状の課題

- FAQ検索のみで制度の構造的説明（法的根拠・算定プロセス・オークション価格推移）ができない
- 解約後いつまで請求が続くかの説明が不十分（問い合わせの6割を占める）
- 年度別単価の変動に関する情報が不足（値上がり問い合わせに対応できない）

### 対象となるZendesk質問パターン

- 「解約したのに容量拠出負担金の請求が来た。最後の請求はいつ？」
- 「2026年4月から容量拠出負担金が値上がりすると書いてあった。詳しく聞きたい」
- 「来年度の容量拠出金単価の概算を教えてほしい」
- 「容量拠出負担金とは何ですか？なぜ請求されるのですか？」
- 「電力を使っていないのに請求されるのはなぜ？」
- 「請求書の容量拠出負担金の対象期間はいつですか？」

### ST-1: 容量拠出金制度の構造的説明ナレッジ追加 ✅

**意図**: OCCTO公式資料に基づく制度の構造的説明を、既存FAQ（DG固有のQ&A）とは別にナレッジチャンクとして追加する。

**完了内容**:
- `guide_容量拠出金制度.yaml` を新規作成（6チャンク）
- 出典: OCCTO「容量拠出金を知ろう！」https://www.occto.or.jp/capacity-market/kyoshutsukin_know/
- チャンク構成:
  1. 制度概要（法的根拠・目的・負担者）
  2. 算定方法（4ステップの算定プロセス）
  3. メインオークション約定価格の推移（2024〜2027年度）
  4. 請求スケジュールと年次精算
  5. 解約後の請求が続く理由（3つの理由 + 最終請求時期の目安）
  6. DGにおける転嫁方法（算定式・変動要因・確認方法）

**対象ファイル**:
- `src/app/services/agent/knowledge/chunks/general/guide_容量拠出金制度.yaml`

### ST-2: knowledge_data.md の容量拠出負担金セクション補強

**意図**: Data Agent が参照する知識ベースの容量拠出負担金セクションを、より実用的な情報で補強する。

**追加すべき内容**:
- オークション約定価格の年度別推移（Data Agent がシミュレーション回答時に参照）
- 解約後の最終請求月の計算ルール（対象月 + 3ヶ月）
- 対象期間の説明（計量期間と請求月のズレ）

**対象ファイル**:
- `src/app/services/agent/prompts/knowledge_data.md` — 既存「容量拠出負担金の算定」セクションを拡充

**設計判断**:
- 専用ツールは作らない。既存の data_agent ツール（`get_demand_contract` で契約容量、`get_endpoint_monthly_charge` で実請求額）+ プロンプト内の計算ルールで LLM が回答を生成する方式
- 年度別単価はナレッジに記載し、年1回手動更新（OCCTO公表後）
