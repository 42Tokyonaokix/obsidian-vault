## 概要

Zendesk問い合わせ33件に対応する解約・切替ウィザードを実装する。解約理由に応じたフロー分岐、解約日→機器撤去日→最終請求日のタイムライン自動生成、必要書類チェックリスト、容量拠出負担金の最終請求月計算を行う。

### 現状の課題

- FAQで手順説明のみ。解約理由ごとの手続き差異が案内できない
- 解約後のタイムライン（撤去日・最終請求日・容量拠出負担金の継続期間）が複雑で、顧客が自力で把握するのは困難
- 「解約したのにまだ請求が来る」という問い合わせが繰り返し発生

### 対象となるZendesk質問パターン

- 「解約の手続きを教えてください」
- 「他社に切り替えたい」
- 「建物取壊しに伴い契約を終了したい」
- 「解約後も請求が来るのはなぜ？」
- 「○月末に解約した場合、最終請求はいつ？」
- 「廃止申請の進捗を知りたい」

### ST-1: 解約フロー分岐ロジックの設計

**意図**: 解約理由に応じて異なる手続き・書類を案内する。

**解約理由と分岐**:

| 解約理由 | 必要手続き | 特記事項 |
|---------|-----------|---------|
| 建物取壊し | 廃止申請 + 機器撤去手配 | 撤去工事の日程調整が必要 |
| 建物売却 | 名義変更 or 解約 | 買主が継続する場合は名義変更 |
| 他社切替 | スイッチング手続き | 切替先が手続きを主導。DGへの連絡は不要な場合も |
| 事業終了 | 廃止申請 | 法人の場合、清算手続きとの連携 |

**実装内容**:
- プロンプトに解約理由ヒアリング→分岐ガイダンスを追加
- 各分岐での必要書類チェックリスト
- エスカレーション時のチケットテンプレート（理由・希望解約日・契約番号）

**対象ファイル**:
- `src/app/services/agent/prompts/` — フロー定義
- `src/app/services/agent/knowledge/chunks/general/` — 解約手続きQ&A

### ST-2: タイムライン自動生成の実装

**意図**: 解約日を起点に、その後の各マイルストーンの日付を自動計算して提示する。

**タイムライン構成**:
1. **解約申込日** → ユーザー入力 or 「今日」
2. **解約日（供給停止日）** → 月末（原則）
3. **機器撤去日** → 解約日後2-4週間（送配電事業者の手配次第）
4. **最終電力量請求** → 解約月 + 1-2ヶ月（検針→請求のリードタイム）
5. **容量拠出負担金の最終請求** → 解約月から最大12ヶ月後（対象期間による）

**実装内容**:
- `generate_cancellation_timeline(cancellation_date, reason)` 関数
- 営業日計算（006-a2 の ST-1 で実装するカレンダー機能を利用）
- 容量拠出負担金の残存請求期間計算

**対象ファイル**:
- `src/app/services/agent/multi_agent/agents/tools/` — 新規ツール or utilities拡張

**依存**: 006-a2 の営業日カレンダー機能

### ST-3: 解約ナレッジチャンクの補強

**意図**: 解約に関するFAQを充実させ、RAGの回答品質を向上させる。

**追加すべきQ&A**:
- 解約後に請求が続く理由（容量拠出負担金、最終精算）
- 解約と他社切替の違い
- 廃止申請の流れと所要期間
- 機器撤去の手配方法
- 解約違約金（契約期間内の解約の場合）

**対象ファイル**:
- `src/app/services/agent/knowledge/chunks/general/` — 既存 `procedures.yaml` に解約関連あり。不足分を補強
- 必要に応じて `faq_解約切替.yaml` を新規作成

### ST-5: 統合テスト・目視レビュー

**テストケース**:

| 質問 | 期待動作 |
|------|---------|
| 「3月末に解約したい」 | 解約理由ヒアリング → タイムライン生成 → 書類案内 |
| 「建物を取壊すので契約終了したい」 | 廃止申請フロー → 撤去手配案内 → タイムライン |
| 「他社に切り替えます」 | スイッチング手続き説明 → DGへの連絡要否 |
| 「解約したのにまだ請求が来る」 | 容量拠出負担金の仕組み説明 → 最終請求月を提示 |
| 「6月末解約の場合、容量拠出の最終請求は？」 | 具体的な月を計算して回答 |

## 依存関係

- ST-1 は独立して実施可能
- ST-2 → 006-a2 ST-1（営業日カレンダー）
- ST-1 + ST-2 → ST-3（フロー確定後にナレッジ補強）

## 前提条件

- 006-a2（請求スケジュール照会）の営業日カレンダーが先に実装されていること
- 容量拠出負担金の対象期間ルール（何月解約→何月まで請求）が確定していること

## スコープ外

- 廃止申請のステータス追跡（DGMまたは送配電事業者のシステム連携が必要）
- 他社切替先の推薦・比較
- 違約金の正確な計算（契約条件により異なるため概算のみ）
